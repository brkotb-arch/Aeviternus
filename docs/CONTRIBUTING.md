# Contributing

Thank you for your interest.

At the current stage the project is primarily maintained by the original author.

Contributions are welcome in the following areas:

- documentation
- bug reports
- testing
- runtime optimization
- architecture discussion

Before opening a Pull Request please create an Issue describing the proposed change.

---

## Development Principles

Aeviternus follows several core development principles:

- modular development
- documented changes
- minimal unnecessary complexity
- maintainable architecture

---

# Before Making Changes

Before contributing, review:

- architecture documentation
- existing modules
- current development roadmap
- project design principles

Understanding the system structure helps maintain consistency across changes.

---

# Code Style

Preferred practices:

- readable and maintainable Python code
- clear naming conventions
- documented behavior
- simple and understandable implementations

Code should prioritize clarity over unnecessary abstraction.

---

# Pull Requests

A good pull request should contain:

- clear explanation of the change
- motivation behind the implementation
- testing information
- possible risks or limitations

Changes should be focused, reviewable, and aligned with the project architecture.

---

# Security

Never commit:

- API keys
- passwords
- private data
- database files
- sensitive configuration files

Security-related information should always remain outside the repository.
